# System Patterns

## Архитектура системы

**Web-приложение с разделением на фронтенд и бэкенд:**

- **Фронтенд:** Next.js (React, TypeScript, TailwindCSS) — SPA/SSR.
- **Бэкенд:** Node.js REST API.
- **База данных:** PostgreSQL.
- **Автоматизации:** N8N — оркестратор интеграций и бизнес-логики.
- **Интеграция:** Avito API (webhooks + polling).
- **Уведомления:** Telegram Bot API.
- **Инфраструктура:** Docker.

## Ключевые доменные сущности

```
Card (Карточка)          — главный объект системы
├── Contact (Контакт)    — покупатель, идентифицируется по Avito User ID
├── Listing (Объявление) — объявление на Avito
├── Chat / Messages      — переписка
├── Order (Заказ)        — заказ через Avito-доставку
├── CardStatus           — текущий статус воронки
├── AIState              — состояние AI (on/off), хранится отдельно
├── Assignee             — ответственный пользователь
└── EventLog             — журнал всех переходов
```

## Воронка продаж (статусы карточки)

```
Входящие → Дожим → Ожидание человека / Человек
         ↓
    Сделка оформлена → К отправке → В пути → Получено клиентом → Исполнено
         ↓
    Отказ / Потеряно  ←←←← возврат во Входящие при новом сообщении
```

**Закрытые финальные статусы:** Исполнено, Потеряно, Отказ  
**Исполнено** — единственный необратимый статус; новое сообщение = новая карточка.

## Паттерны проектирования

### Расширяемые статусы
Статусы НЕ хардкодятся как enum-константы с жёсткой логикой. Модель должна поддерживать добавление новых статусов и переходов без рефакторинга ядра.

### Event Log (журнал событий)
Каждый переход фиксируется с источником:
- `AI` — AI-агент;
- `user` — действие пользователя;
- `avito` — событие из Avito API;
- `n8n` — автоматизация N8N;
- `system` — таймаут или системное событие.

### Раздельное хранение состояний карточки
Карточка хранит отдельно:
- `status` — текущий статус воронки;
- `ai_enabled` — состояние AI (boolean);
- `assignee_id` — ответственный;
- `reason_human_required` — почему нужен человек;
- `decline_reason` — причина отказа;
- `is_deal` — флаг оформленной сделки;
- данные заказа и доставки.

### AI on/off логика
- AI on → человек не пишет вручную; AI off → статус «Человек», ручной ввод разблокирован.
- При включении AI обратно — карточка переходит в «Дожим».
- Состояние AI хранится отдельно от статуса карточки.

### Создание карточки
Алгоритм при входящем сообщении:
1. Определить контакт по Avito User ID (создать если нет).
2. Проверить активную/возвратную карточку по `(contact, listing)`.
3. Если есть активная → добавить сообщение в неё.
4. Если «Потеряно»/«Отказ» → вернуть карточку во «Входящие».
5. Если «Исполнено» → создать новую карточку.
6. Если нет совсем → создать новую карточку.

## N8N — основные сценарии

N8N является оркестратором интеграций. Через N8N проходят:
- Получение событий из Avito (вебхуки / polling).
- Обработка входящих сообщений и создание карточек.
- Вызов AI-логики и обновление карточек.
- Обновление статусов заказа / доставки.
- Синхронизация объявлений.
- Telegram-уведомления.
- Таймауты → перевод в «Потеряно».
